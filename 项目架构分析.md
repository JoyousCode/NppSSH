# 三、你的架构是什么设计模式？

## **答案：外观模式（Facade Pattern）+ 中介者模式（Mediator Pattern）**

### 1. 外观模式（Facade）

- **SSHWindow 对上层隐藏底层复杂系统**

- 外部只调用简单接口，不用管内部多少模块

- 符合：**提供统一入口，简化调用**

### 2. 中介者模式（Mediator）

- **所有模块互不通信，只与 SSHWindow 通信**

- 模块之间零耦合

- 符合：**降低网状依赖，变成星型结构**

# 五、优点（非常强，大型项目标准架构）

## 优点 1：**完全解耦，模块之间零依赖**

- SSHConnection 不知道 SSHLog 存在

- SSHPanel 不知道 SSHConnection 存在

- 所有模块只依赖 SSHWindow

- **删除任何模块，不影响其他模块编译**

## 优点 2：**易于维护、易于扩展**

- 加新功能，只改中转，不改所有调用处

- 统一修改，一处修改全插件生效

## 优点 3：**便于管理全局变量 & 全局状态**

- 连接状态、面板数量、窗口句柄、NppData 全部集中管理

- 不会出现变量到处定义的混乱

## 优点 4：**调用链路清晰，不会出现循环依赖**

- 传统架构容易出现：

    - A引用B

    - B引用A

    → 编译报错

- 你的架构：

    - A → 中转 → B

    - B → 中转 → A

    **完全安全**

## 优点 5：**符合大型软件、插件系统标准架构**

Notepad++插件、Chrome插件、VS插件 都这么做。

# 六、缺点（真实存在，我直接告诉你）

## 缺点 1：SSHWindow 会变成“上帝类”

功能越多，SSHWindow 代码越长

→ 未来会达到 1000～2000 行

## 缺点 2：调用链路变长

A → 中转 → B

比 A→B 多一层跳转

## 缺点 3：新手阅读代码需要多跳一层

不如直接调用直观

## 缺点 4：所有模块必须依赖 SSHWindow

如果 SSHWindow 出错，整个插件崩溃
